Step Execution Model\n   408→\n   409→### 3.1 Execution Engine\n   410→\n   411→The orchestrator is **not** a new tool. It runs inside auto-delegate's planning phase. When auto-delegate recognizes a pipeline trigger, it switches from ad-hoc planning to pipeline execution mode.\n   412→\n   413→```\n   414→User message → Mode Router → Task mode → Auto-delegate triage\n   415→  → T0: direct execution\n   416→  → T1: lightweight plan + single agent\n   417→  → T1/T2 + pipeline match: PIPELINE EXECUTION MODE ← this spec\n   418→  → T2 (no pipeline match): full planning phase\n   419→```\n   420→\n   421→### 3.2 Step Sequencing Algorithm\n   422→\n   423→```\n   424→1. Parse pipeline YAML\n   425→2. Build dependency graph from step.input.from references\n   426→3. Topological sort for execution order\n   427→4. Identify parallel groups: steps with parallel_with or steps whose\n   428→   inputs are all satisfied simultaneously\n   429→5. Execute groups sequentially, steps within group in parallel (tmux)\n   430→6. After each step: check gate, route on gate result\n   431→```\n   432→\n   433→For the XHS pipeline, the execution plan becomes:\n   434→\n   435→```\n   436→[sequential] spec\n   437→[sequential] write\n   438→[sequential] render\n   439→[parallel]   qc + fact_check    ← both depend on write, independent of each other\n   440→[sequential] publish            ← depends on qc AND fact_check passing\n   441→```\n   442→\n   443→### 3.3 Agent Spawning per Step\n   444→\n   445→Each step spawns a fresh agent. The orchestrator:\n   446→\n   447→1. Reads the step's `agent` block\n   448→2. Constructs a `claude -p` command with:\n   449→   - `--model` from `agent.model`\n   450→   - `--max-budget-usd` from `agent.budget_usd`\n   451→   - Prompt assembled from: persona + input artifacts + output contract\n   452→3. For parallel steps: spawns via tmux, waits with `tmux wait-for`\n   453→4. For sequential steps: spawns inline (subagent) or via tmux\n   454→\n   455→```bash\n   456→# Example: spawning the QC step\n   457→tmux new-session -d -s pipeline-xhs-qc\n   458→tmux send-keys -t pipeline-xhs-qc \\\n   459→  'claude -p --model claude-sonnet-4-6 --max-budget-usd 0.50 \\\n   460→    --dangerously-skip-permissions --output-format json \\\n   461→    \"You are an independent QC auditor. The orchestrator:\n   477→\n   478→1. After step completes: reads the output artifact path\n   479→2. Verifies output contract (script checks or file existence)\n   480→3. Passes artifact path (not content) to downstream steps' input assembly\n   481→\n   482→This is stateless -- if the orchestrator crashes and restarts, it can check which artifacts exist and resume from the last incomplete step.\n   483→\n   484→## 4. Gate Protocol\n   485→\n   486→### 4.1 Gate Types\n   487→\n   488→| Type | Mechanism | When to Use |\n   489→|------|-----------|-------------|\n   490→| `script` | Shell script exits 0 (pass) or 2 (block) | Format checks, existence checks, test suites |\n   491→| `agent` | Reviewer agent evaluates output against contract | Semantic quality, QC audits, design reviews |\n   492→| `both` | Script runs first; if pass, agent reviews | High-stakes: format + semantic |\n   493→\n   494→### 4.2 Gate Input/Output Contract\n   495→\n   496→Every gate receives:\n   497→- The step's output artifact\n   498→- The step's output contract (as evaluation criteria)\n   499→- The step's input artifacts (for context)\n   500→\n   501→Every gate produces:\n   502→- **Verdict:** PASS / FAIL\n   503→- **Details:** If FAIL, structured list of issues\n   504→- **Suggested action:** retry (same agent), escalate (better model), ask_user, abort\n   505→\n   506→### 4.3 Gate Enforcement\n   507→\n   508→Gates are not advisory. A FAIL verdict blocks the pipeline. The orchestrator cannot skip a failed gate.\n   509→\n   510→```\n   511→Gate result → PASS → next step\n   512→Gate result → FAIL → check on_fail:\n   513→  retry     → re-run same step (up to max_retries), injecting gate feedback as additional input\n   514→  escalate  → re-run step with upgraded model (sonnet → opus) or spawn expert agent\n   515→  ask_user  → HITL relay: write question to records/QUESTION.md, wait for answer\n   516→  abort     → stop pipeline, save state, notify user\n   517→```\n   518→\n   519→### 4.4 Relationship to Existing Hooks\n   520→\n   521→Pipeline gates and CC hooks are complementary:\n   522→\n   523→| Layer | Mechanism | Scope |\n   524→|-------|-----------|-------|\n   525→| **CC PreToolUse hook** | Blocks individual tool calls (Write, Bash, etc.) | Within a single agent |\n   526→| **CC PostToolUse hook** | Validates tool output | Within a single agent |\n   527→| **Pipeline gate** | Blocks pipeline progression between steps | Between agents |\n   528→\n   529→Pipeline gate scripts can reuse existing hook scripts. For example, `pre-publish-xhs.sh` already exists as a gate script -- the pipeline just calls it at the right point.\n   530→\n   531→## 5. Failure Handling Matrix\n   532→\n   533→### 5.1 Failure Categories\n   534→\n   535→| Category | Signal | Example | Default Response |\n   536→|----------|--------|---------|-----------------|\n   537→| **Agent error** | Non-zero exit, no artifact | Agent crashed, budget exhausted | Retry once, then escalate |\n   538→| **Contract violation** | Artifact exists but fails checks | QC verdict is REVISE, tests fail | Retry with feedback |\n   539→| **Gate block** | Gate script exits 2 | Format check failed | Retry step |\n   540→| **External failure** | MCP tool error, network timeout | XHS publish API down | Ask user (not retryable) |\n   541→| **Budget exceeded** | Step cost > budget_usd | Complex task needs more turns | Escalate (more budget) or ask user |\n   542→| **Pipeline-level** | 3+ steps failed in sequence | Systemic issue | Abort + save state |\n   543→\n   544→### 5.2 Escalation Ladder\n   545→\n   546→```\n   547→Attempt 1: Same agent, same model, inject failure feedback\n   548→Attempt 2: Same agent, same model, different strategy (from failure analysis)\n   549→Attempt 3: Escalate model (sonnet → opus)\n   550→Attempt 4: Spawn expert agent for diagnosis\n   551→Attempt 5: Ask user (decision gap)\n   552→```\n   553→\n   554→This mirrors the existing auto-delegate gap classification:\n   555→- Retry = capability within reach, just needs another attempt\n   556→- Escalate model = ability gap (need stronger reasoning)\n   557→- Spawn expert = knowledge gap (need diagnosis, not more attempts)\n   558→- Ask user = decision gap (only interrupt for this)\n   559→\n   560→### 5.3 Retry with Feedback Injection\n   561→\n   562→When a step is retried after gate failure, the orchestrator injects the gate's feedback into the new agent's prompt:\n   563→\n   564→```\n   565→=== PREVIOUS ATTEMPT FAILED ===\n   566→Gate verdict: FAIL\n   567→Issues found:\n   568→1.
